### À propos du projet
Ce projet est né d'un besoin simple, mais exigeant : expérimenter la programmation assistée par IA pour mieux comprendre ce que cela implique concrètement.
Pas comme un exercice académique.
Pas comme un « side project » jetable.
Mais comme un véritable terrain d'expérimentation, pensé dès le départ pour produire du code compréhensible, testable, maintenable — et partageable, afin de voir comment mon expérience peut être déterminante (ou pas) pour produire du code au-dessus de la moyenne.
Le code ainsi produit sera relu par des développeurs expérimentés (comme grasfter) pour évaluer sa qualité.
L'objectif est aussi de mieux interagir avec les développeurs à l'ère de l'IA.

### Une expérimentation ambitieuse
Mon objectif est d'expérimenter :
- la programmation orientée objet
- dans un contexte web complet (front-end et back-end)
- avec des langages et outils modernes
- en appliquant, même à petite échelle, les bonnes pratiques du software craftsmanship
Pour y parvenir, j'utilise une IA comme partenaire de travail, non pas pour coder « à ma place », mais pour jouer deux rôles très précis :
- produire du code sous ma supervision
- expliquer en détail ce code, son architecture et ses choix de conception, afin que je comprenne ce qui est construit et puisse orienter les choix
L'IA est ici un outil d'expérimentation qui me permet d'évaluer comment mon expérience de Product Manager et mon passé de développeur peuvent influencer la qualité du code produit.

### Les principes non négociables du projet
Même pour un projet personnel, même pour apprendre, je considère que le minimum est :
- TDD : les tests guident le code
- CI/CD : le code est versionné, testé et déployable
- un historique clair sur GitHub
- un déploiement rapide dans un environnement accessible à des utilisateurs réels (testeurs, amis, tiers)
- des itérations courtes et du feedback rapide
Le code n'est pas écrit « pour moi seul », mais pour être :
- lu
- compris
- critiqué
- amélioré

### Un accent fort sur l'expression du métier
Je suis particulièrement attaché à la capacité du code à exprimer l'intention métier.
C'est pourquoi le projet s'appuie aussi sur :
- des scénarios BDD en Gherkin
- un langage compréhensible par des non-développeurs
L'objectif est simple : qu'une personne extérieure puisse retrouver, dans le code source, l'intention initiale du produit.

### Pourquoi ce projet peut vous intéresser
Si vous êtes recruteur, développeur ou tech lead, ce projet n'est pas là pour impressionner par sa taille ou sa complexité.
Il est là pour montrer :
- une démarche d'expérimentation sérieuse sur la programmation assistée par IA
- une culture de la qualité logicielle appliquée même en expérimentation
- une capacité à évaluer et orienter la production de code avec méthode
- une exigence souvent réservée aux projets professionnels, appliquée ici à une expérimentation personnelle
C'est à la fois un laboratoire d'expérimentation, un outil de compréhension des enjeux de l'IA dans le développement, et une démonstration de comment un Product Manager peut interagir efficacement avec des développeurs à l'ère de l'IA.
Ces bonnes pratiques sont formalisées dans une Definition of Done (DOD) structurée par thème, du plus simple au plus complexe.
