# Unification de l'architecture des images

## Réorganisation des images avec sous-dossiers par type

### Prompt
Question d'architecture sur l'unification des images. Actuellement, certaines images sont dans `public/images/` (pratique courante Next.js) et d'autres dans `data/images/` (demande explicite pour les images "de la base de données" CMS). Proposition d'unifier en créant des sous-dossiers dans `data/images/` :
- `data/images/json/` pour les images référencées dans les JSON
- `data/images/md/` pour les images référencées dans les Markdown
- `data/images/static/` pour les images statiques (Logo, Photo du header)

Dans les fichiers, on ne précise plus que le nom du fichier (sans chemin complet).

### Résultat technique
Unification complète de l'architecture des images :

**1. Structure des dossiers** :
- Toutes les images déplacées de `public/images/` vers `data/images/`
- Création de 3 sous-dossiers : `json/`, `md/`, `static/`
- Le dossier `public/images/` est maintenant vide

**2. API Route** :
- Nouvelle route : `/api/images/[type]/[filename]`
- Types acceptés : `json`, `md`, `static`
- Validation du type et service depuis `data/images/{type}/`

**3. Fonctions utilitaires** (`utils/imagePath.ts`) :
- `getJsonImagePath(filename)` → `/api/images/json/${filename}`
- `getMdImagePath(filename)` → `/api/images/md/${filename}`
- `getStaticImagePath(filename)` → `/api/images/static/${filename}`

**4. Composants modifiés** :
- `DomaineDeCompetences.tsx` : utilise `getJsonImagePath()` pour les images de compétences
- `Temoignages.tsx` : utilise `getJsonImagePath()` pour les photos
- `AboutSiteContentRenderer.tsx` : utilise `getMdImagePath()` pour les images Markdown
- `aboutSiteReader.ts` : construit automatiquement le chemin MD pour les images Markdown
- `headerImages.ts` : utilise `getStaticImagePath()` pour Logo et Photo

**5. Configuration et tests** :
- `next.config.ts` : headers de cache mis à jour pour `/api/images/:type/:path*`
- Tests unitaires mis à jour avec les nouveaux chemins

**6. Format dans les fichiers** :
- **JSON** : `"src": "Logo.png"` (juste le nom de fichier)
- **MD** : `![alt](Logo.png)` (juste le nom de fichier, sans `/api/images/`)

**Avantages** :
- Séparation claire entre images JSON, MD et statiques
- Réduction de la redondance (pas de chemin complet dans les fichiers)
- Flexibilité (changement de chemin centralisé dans le code)
- Cohérence : contenu et images au même endroit (approche CMS)

Build réussi. Tous les tests passent.

## Refactorisation de la structure des domaines de compétences

### Prompt

### Résultat technique
**Architecture par références implémentée** :

**1. Modifications JSON** (`data/ingenierie-logiciel.json`) :
- Remplacement du contenu complet des domaines par des références :
  - `"ref": "developpeur"`
  - `"ref": "experience-en-equipe"`

- Architecture plus modulaire et maintenable
- Prépare une possible centralisation des domaines de compétences dans un fichier dédié
